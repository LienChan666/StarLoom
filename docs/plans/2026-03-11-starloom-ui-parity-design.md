# StarLoom UI 完整复刻设计

## 目标

在保留重写工作树 `Ui/Pages` 入口结构的前提下，把当前最小骨架版 UI 恢复为与原项目一致的主窗口、首页、设置页和状态悬浮窗，并且恢复原版交互能力，而不把整体架构退回旧的 `Automation` 模式。

## 范围

- 完全复刻原版主窗口尺寸、标签文案和布局层级。
- 完全复刻首页三块区域：
  - 左侧控制面板
  - 右上搜索面板
  - 右下已选兑换列表
- 完全复刻设置页四个分区：
  - 商店设置
  - 制作点设置
  - 购买设置
  - 界面设置
- 完全复刻状态悬浮窗的样式、状态文案和停止入口。
- 恢复原版本地化键集与界面文案切换行为。
- 恢复原版共享 UI 表现层，包括布局度量、面板样式、货币标签等。

## 不做的事

- 不把重写分支的 UI 重新接回旧的全局静态入口。
- 不额外引入 ViewModel 层、适配层或兜底层。
- 不为复刻 UI 而回退 `WorkflowTask` 为旧的 `WorkflowOrchestrator` 架构。
- 不借这次 UI 改造顺手重做视觉设计。

## 设计原则

### 1. 保留新入口，恢复旧体验

重写分支继续保留：

- `Ui/MainWindow.cs`
- `Ui/Pages/HomePage.cs`
- `Ui/Pages/SettingsPage.cs`

这三个文件作为新架构中的页面入口与组装点，不恢复旧项目的目录入口方式。

在此基础上恢复组件层：

- `Ui/Components/Home/*`
- `Ui/Components/Settings/*`
- `Ui/Components/Shared/*`

页面负责组装，组件负责布局、交互和样式细节。

### 2. UI 只消费现有业务状态

UI 仍然只通过以下对象获取状态和发起入口：

- `WorkflowTask`
- `ConfigStore`
- UI 所需的最小只读数据入口

UI 不直接调用底层游戏行为，不直接调用 IPC，不新增中间状态层。

### 3. 交互复刻优先于代码复用

本次目标是“在新结构中重实现原版 UI”，不是简单把旧文件复制过来。因此允许参考旧项目的具体布局、文案、输入项和行为，但最终实现必须落在重写分支当前结构下。

## 组件结构

### 主窗口

`Ui/MainWindow.cs` 保持轻量，负责：

- 设定与原版一致的默认窗口尺寸
- 使用本地化标签绘制 `Home` 与 `Settings` 顶层标签
- 组装 `HomePage` 与 `SettingsPage`

### 首页

`Ui/Pages/HomePage.cs` 负责组装原版首页布局：

- 左侧 `HomeControlPane`
- 右上 `SearchPane`
- 右下 `SelectedItemsPane`

首页布局继续通过 `LayoutMetrics` 控制宽高与区域比例，保证与原版一致。

#### HomeControlPane

负责：

- 显示总状态
- 显示当前清单 ID
- 编辑清单 ID
- 启动主流程
- 停止流程
- 启动“仅提交”
- 启动“仅购买”
- 在缺少购买物品配置时显示提示

它只调用 `WorkflowTask` 暴露的入口与状态，不承载业务判断。

#### SearchPane

负责：

- 显示搜索输入框
- 显示商店物品加载状态
- 按名称筛选工票物品
- 显示结果表格
- 执行“加入兑换列表”

搜索数据来源于重写分支内可用的商店候选物品数据。如果当前重写实现中缺少该入口，则新增最小必要的只读提供方式，但不加包装层。

#### SelectedItemsPane

负责：

- 显示当前已选兑换列表
- 显示名称、货币、单价、目标数量、顺序、操作列
- 编辑数量
- 上下移动顺序
- 删除项目
- 修改后立即保存配置

### 设置页

`Ui/Pages/SettingsPage.cs` 负责组装原版四个设置分区，内部通过标签页承载：

- `ShopSettingsCard`
- `CraftPointSettingsCard`
- `PurchaseSettingsCard`
- `DisplaySettingsCard`

#### ShopSettingsCard

负责收藏品商店相关选择与显示。

#### CraftPointSettingsCard

负责：

- 刷新返回点
- 显示返回点列表状态
- 选择默认返回点
- 显示当前返回点状态

如果当前重写分支没有提供足够的返回点只读数据，则补足该能力，但只补 UI 所需最小入口。

#### PurchaseSettingsCard

负责：

- 设置购买完成动作
- 设置工票预留
- 设置背包保护阈值

#### DisplaySettingsCard

负责：

- 状态悬浮窗开关
- 界面语言切换
- 配置保存与语言资源刷新

### 状态悬浮窗

`Ui/StatusOverlay.cs` 复刻原版的：

- 圆角与边框样式
- 背景色
- 状态点
- 状态文案
- 分隔线
- 停止按钮

它继续只依赖 `WorkflowTask` 与 `ConfigStore`。

## 共享表现层

恢复以下共享表现层文件：

- `Ui/Components/Shared/LayoutMetrics.cs`
- `Ui/Components/Shared/GamePanelStyle.cs`
- `Ui/Components/Shared/ScripShopUiHelpers.cs`

其中：

- `LayoutMetrics` 负责首页与设置页的区域尺寸计算
- `GamePanelStyle` 负责颜色、边框、按钮、分隔线和状态点
- `ScripShopUiHelpers` 负责物品货币标签的统一显示

## 本地化

恢复并对齐原版 `Resources/Localization/zh.json` 与 `en.json` 所需的完整 UI 键集，包括：

- 顶层标签
- 首页控制面板
- 搜索面板
- 已选列表
- 设置页四个分区
- 状态悬浮窗
- 流程状态文本

重写分支不再允许在页面中写死 `"Home"`、`"Settings"`、`"Stop"` 这类文案。

## 数据流

UI 数据流保持简单直接：

- `WorkflowTask` 提供流程状态、可执行入口、错误信息
- `ConfigStore.pluginConfig` 提供所有可编辑配置
- 页面从重写分支的现有业务对象读取只读展示数据
- UI 修改配置后立即 `Save()`

不增加草稿态，不增加临时同步层，不增加回退逻辑。

## 验收标准

### 结构一致

- 主窗口默认尺寸与原版一致
- 首页恢复三块区域布局
- 设置页恢复四个分区布局
- 悬浮窗样式与原版一致

### 交互一致

- 开始、停止、仅提交、仅购买按钮状态一致
- 搜索物品、加入列表、编辑数量、排序、删除可用
- 商店、返回点、购买、界面设置均可编辑并保存
- 状态悬浮窗可显示状态并停止流程
- 界面语言切换行为与原版一致

### 架构边界一致

- 保留 `Ui/Pages` 页面入口
- 不回退为旧的全局静态 UI 组织方式
- 不增加新的中间抽象层
- UI 只消费任务与配置状态

- 悬浮窗样式和停止入口
- 中英文文案切换
